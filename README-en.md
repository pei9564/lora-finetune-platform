# Finetune Platform — End-to-End LoRA Training & Experiment Management Platform

This project is a production-grade LoRA (Low-Rank Adaptation) fine-tuning platform designed to integrate the entire lifecycle of model development—from data validation and asynchronous training to experiment tracking and model governance.

This project was awarded **Honorable Mention** in the **"Generative AI Category" of the 2025 iThome Ironman Competition**.

* **Project Theme**: Building an AI Fine-tuning Platform: 30 Days of Practical Notes from System Design to AI Collaboration.
* **Development Model**: This project deeply implements the AI-Augmented Development (AAD) paradigm. Approximately 80% of the core code and configurations were co-generated by **ChatGPT-4o** and **Cursor**. Architectural decisions, prompt sets, and human-AI collaboration notes are detailed in the Ironman competition article series.

## Technical Architecture

The platform adopts a decoupled, cloud-native layered architecture to ensure the isolation of compute-intensive tasks (LLM Fine-tuning) from the management plane (Control Plane), catering to production-level scalability requirements:

* **Control Plane (Access & Control)**:
  * **API Layer**: Built with FastAPI and exposed via Kubernetes Services. It handles task submission, JWT authentication, RBAC (Role-Based Access Control), and task status distribution.

* **Orchestration Layer (Task Scheduling)**:
  * **Task Orchestration**: Implements asynchronous task scheduling via Celery and Redis. Redis serves as the Message Broker to ensure reliable task distribution. It features a robust polling mechanism and error-retry logic designed for long-running training jobs.

* **Execution Layer (Computation Engine)**:
  * **Fine-tuning Engine**: The training scripts support automatic hardware detection. It prioritizes NVIDIA CUDA in Linux environments and automatically enables Apple MPS acceleration in macOS (M3/M4) environments.
  * **Kubernetes Pod Resources Limit**: Strictly manages computational resources to ensure the stability of training tasks and the overall cluster.

* **Governance Layer (Storage & Governance)**:
  * **MLOps Integration**: Integrates MLflow Tracking for real-time logging of experiment metrics and artifacts. It utilizes the Model Registry to achieve experiment versioning and lifecycle stage management.

## Technical Features & Implementation

### 1. Experiment Tracking & Model Governance (MLOps)

The system treats every training session as a complete experimental entity rather than just producing weight files:

* **Automated Model Card**: Upon completion of each training session, the system automatically generates standardized JSON metadata, serving as the foundation for model governance and semantic recommendation.
* **Lifecycle Management**: Integrates MLflow Registry API to support model stage transitions—`Staging`, `Production`, and `Archived`—ensuring environment consistency.

#### Model Card JSON Example

```json
{
  "id": "chinese-sentiment-v1",
  "name": "Chinese Sentiment Model",
  "base_model": "bert-base-chinese",
  "language": "zh",
  "task": "sentiment",
  "description": "Fine-tuned BERT for Chinese movie reviews",
  "metrics": { "accuracy": 0.89 },
  "tags": ["Chinese", "Sentiment Analysis", "BERT"],
  "embedding": [0.1, 0.2, -0.05, 0.3]
}
```

### 2. Data Validation & Semantic Recommendation

* **Schema-Driven Data Validation**: Performs format checks and distribution analysis on uploaded JSONL data before tasks enter the queue, preventing invalid tasks from wasting computational resources.
* **Semantic Recommendation System**: Leverages the `embedding` vector in the Model Card to implement semantic similarity search. Users can describe their needs in natural language, and the system automatically recommends the most suitable Adapter.

### 3. Observability

* **Prometheus Exporter**: Monitors custom metrics including `task_queue_length` (backlog monitoring), `task_duration_seconds` (performance analysis), and `task_failure_total`.
* **Grafana Dashboard**: Provides visual panels for real-time monitoring of worker node CPU/Memory loads and training task success trends.

## Deployment & Automation (CI/CD)

The project utilizes Helm as the deployment standard:

* **Helm Chart Structure**: Separates `values.yaml` (Dev) and `values.prod.yaml` (Production) configurations. It uses a StatefulSet for Redis data persistence.
* **GitHub Actions Workflow**:
* **CI**: Automatically executes Pytest unit tests and Linting for code quality.
* **CD**: Validates templates via Helm Dry-run and automatically builds Docker images tagged with version labels (day-tag) for the registry.

## Quick Start Example

### Environment Configuration

Create a `.env` file in the root directory:

```env
REDIS_URL=redis://localhost:6379/0
MLFLOW_TRACKING_URI=http://localhost:5000
JWT_SECRET=your_secret_key
```

### Deploy to Kubernetes via Helm

```bash
helm install finetune-platform ./charts/finetune-platform -f ./charts/finetune-platform/values.yaml
```
